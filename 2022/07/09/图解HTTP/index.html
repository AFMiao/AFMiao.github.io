了解Web及网络基础

本章将介绍Web是建立于何种技术之上, 以及HTTP协议是如何诞生并发展的.

使用HTTP协议访问Web

从我们在浏览器中输入URL到Web网页被显示出来, 这之间都发生了什么?

浏览器将根据Web浏览器地址栏中指定的URL从Web服务器端获取文件资源(resource)等信息, 从而显示出Web页面.

像这种通过发送请求获取服务器资源的Web浏览器等均可被称作客户端(client).

Web使用一种名为HTTP(HyperText Transfer Protocol, 超文本传输协议)的协议作为规范, 以完成从客户端到服务器端等一系列运作流程.

HTTP的诞生

很久很久以前, CERN的Doctor Time Berners-Lee提出了一种能够让远隔两地的研究者们共享知识的设想. 最初设想的基本理念是: 借助多文档之间相互关联行程的超文本(HyperText), 练成可相互参阅的WWW(World Wide Web, 万维网).

现在已经有了三项WWW构建技术, 分别为将SGML(Standard Generalized Markeup Language, 标准通用标记语言)作为页面的文本标记语言的HTML、作为文档传递协议的HTTP、以及指定文档所在地址的URL(Uniform Resource Locator, 统一资源定位符).

在当时, HTTP协议还只是被用于解决Web框架中的文件传输额外难题; 但现在, HTTP已经突破了Web框架的限制, 被应用到了各种场景中.

网络基础TCP/IP

为了理解HTTP, 我们有必要事先了解一下TCP/IP协议族.

通常使用的网络(包括互联网)都是在TCP/IP协议族的基础上运作的, 而HTTP输入它内部的一个子集.

这里我们只介绍理解HTTP所需掌握的基本TCP/IP的知识.

TCP/IP协议族

要实现计算机与网络设备间的相互通信, 双方就必须基于相同的方法. 不同的硬件和之间的通信需要用到一种规则, 而这种规则就被称为协议(protocol), 而将与互联网相关联的协议全部集合起来的总和就是TCP/IP.

TCP/IP的分层管理

TCP/IP协议族中最重要的一点就是分层. 按层次, TCP/IP协议族可被划分为以下四层, 即应用层、传输层、网络层和数据链路层.

将TCP/IP层次化是有好处的(类似于面向对象设计), 在此不过多论述. TCP/IP协议族中各层的作用如下:

应用层

应用层决定了向用户提供应用服务时通信的活动. TCP/IP协议族中预存了各类通用的应用服务. 比如FTP(File Transfer Protocol, 文件传输协议)和DNS(Domain Name System, 域名系统)服务就是其中两类; 本书介绍的重点HTTP也在该层中.

传输层

传输层对上层应用层, 提供处于网络连接中德两台计算机之间的数据传输.

在传输层中有两个性质不同的协议: TCP(Transmission Control Protocol, 传输控制协议)以及UDP(User Data Protocol, 用户数据报协议).

网络层

网络层又名又名网络互连层, 用于处理在网络上流动的数据包. 数据包是网络传输的最小数据单位. 该层规定了通过怎样的路径(即所谓的传输路线)到达对方计算机, 并把数据包传送给对方.

在与对方计算机之间通过多态计算机或网络设备进行传输时, 网络层所起到的作用就是在众多的选项内选择一条传输路线.

链路层

链路层又名数据链路层、网络接口层, 用于处理连接网络的硬件部分. 硬件上的范畴径在链路层的作用范围之内.

TCP/IP通信传输流

在利用TCP/IP协议族进行网络通信时, 会通过分层顺序与对方进行通信. 发送端从应用层往下走, 接收端则从应用层往上走. 以HTTP举例:

  1. 首先, 作为发送端的客户端在应用层(HTTP协议)发出一个想看某个Web页面的HTTP请求;
  2. 其次, 为了传输方便, 在传输层(TCP协议)把从应用层接收到的数据(HTTP请求报文)进行分割, 并在各个报文上打上标记䆨及端口号后转发给网络层;
  3. 在网络层(IP协议)处, 增加作为通信目的地的MAC地址后转发给链路层.

如此, 法网网络的通信请求就准备齐全了.

接收端的服务器在链路层接收到数据, 并按序向上层发送, 直到应用层. 当传输到应用层, 才能算真正接收到客户端发送过来的HTTP请求.

在发送端进行层间的数据传输时, 数据每经过一层都会被打上一个该层所属的首部信息. 相反, 在接收端进行层间的数据传输时, 每经过一层, 它都会去掉数据中对应的首部信息. 这种将数据信息包装起来的做法被称作封装(encapsulate).

与HTTP关系密切的协议: IP、TCP和DNS

负责传输的IP协议

IP(Internet Protocol)协议位于网络层. “网路协议”的名字听起来多少有些唬人, 但事实是: 几乎所有的网络系统都会使用IP协议. TCP/IP协议族的名称中有一半是”IP”, 其重要性可见一斑.

IP协议的作用是将各种数据包传送给对方. 要保证数据包的确被传送到对方处, 有很多条件需要满足. 而其中最重要的两个条件即为IP地址和MAC地址(Media Access Control Address). 其中, IP地址指明了节点被分配到的地址, 而MAC地址是指网卡所属的固定地址. IP地址可以和MAC地址进行配对. IP地址可变换, 而MAC地址基本上不会改变.

使用ARP协议凭借MAC地址进行通讯

IP间的通信依赖MAC地址. 在网络上, 通信双方同时位于同一局域网(LAN)内的情况并不多见, 通常需要经过多台计算机和网络设备中转才能连接到对方. 而在中转时, 设备会利用下一站中转设备的MAC地址来搜索下一个中转目标. 此时, 设备会采用ARP协议(Address Resolution Protocol), 一种用于解析地址的协议. 利用该种协议, 设备就可以通过通信方的IP地址反查出对应的MAC地址.

没有人能够全面掌握互联网中的传输状况

在到达通信目标前的中转过程中, 那些计算机和路由器等网络设备只能获悉很粗略的传输路线. 这种机制被称作路由选择(routing). 无论哪台计算机或网络设备都无法全面掌握互联网中的细节.

确保可靠性的TCP协议

TCP协议位于传输层, 负责提供可靠的字节流服务.

所谓字节流服务(Byte Stream Service)是指, 为了方便传输, 将大块数据分割成以报文段(segment)为单位的数据包进行管理. 而可靠的传输服务是指能够把数据准确可靠地传给对方. 总而言之, 为了更容易传输大数据, TCP协议将数据分割为数据包, 且TCP协议能够确认数据是否最终送达对方.

为了数据能够到达目标

为了准确无误第将数据送达目标处, TCP协议采用了著名的三次握手策略(three-way handshaking). 用TCP协议把数据包送出去后, 它一定要确认对方受到了数据. 握手过程中使用了TCP的标志(flag)——SYN(synchronize)和ACK(acknowledgement).

首先, 发送端会发送一个带有SYN标志的数据包给对方; 在接收端收到后, 它会回传一个带有SYN/ACK标志的数据包, 表示自己收到了信息; 最后, 发送端再发送一个带有ACK标志的数据包, 表示”握手”结束.

若握手过程中出现间断, TCP协议就会再次以相同的顺序发送相同的数据包.

负责域名解析的DNS服务

DNS(Domian Name System)服务是与HTTP协议一样位于应用层的协议. 它提供域名到IP地址之间的解析服务.

一个计算机即可被赋予IP地址, 也可被赋予主机名和域名, 如afmiao.github.io

通常来说, 用户会使用主机名或域名来访问对方的计算机, 而不是直接通过难以理解的IP地址访问; 但要让计算机去理解名称, 就相对比较困难了, 毕竟计算机还是更擅长处理数字.

为了解决上述问题, DNS服务应运而生. DNS服务通过域名查找IP地址, 或逆向通过IP地址查找域名服务.

各种协议与HTTP协议的关系

URI和URL

与URI(统一资源标识符)相比, 我们更熟悉URL(Uniform Resource Locator, 统一资源定位符). URL其实就是Web浏览器访问Web页面时使用的网页地址, 如https://afmiao.github.io/.

统一资源标识符

URI是Uniform Resource Identifier的缩写. RFC2396对这三个单词分别进行了如下定义:

  • Uniform: 规定统一的格式可方便处理多种不同类型的资源, 免于根据上下文环境推断资源指定的访问方式之苦, 也让添加新的协议方式变得更为容易.
  • Resource: 所谓资源指的是可标识的任何东西. 除了文档文件、图像或服务等能够区别于其他类型的, 全部都可作为资源. 另外, 资源不仅可以是单一的, 也可以是多数的集合体.
  • Identifier: 表示可标识的对象, 也可称为标识符.
